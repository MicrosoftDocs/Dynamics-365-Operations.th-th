# <a name="set-up-and-install-regression-suite-automation-tool-tutorial"></a>ตั้งค่าและติดตั้งบทช่วยสอนของ Regression Suite Automation Tool
หัวข้อนี้เป็นบทช่วยสอนที่ช่วยให้คุณได้รับการตั้งค่าและเริ่มต้นด้วย RSAT และเครื่องมือที่เกี่ยวข้องกับการใช้ RSAT 

[!include [banner](../includes/banner.md)]

> [!NOTE]
> ใช้เครื่องมือเบราว์เซอร์อินเทอร์เน็ตของคุณเพื่อดาวน์โหลดและบันทึกหน้านี้ในรูปแบบ PDF 

## <a name="key-concepts"></a>แนวคิดหลัก

- ทำความเข้าใจการตั้งค่าของการติดตั้งและข้อกำหนดเบื้องต้นที่จำเป็นสำหรับแอพลิเคชันต่างๆ ที่สนับสนุน Regression Suite Automation Tool (RSAT)
- ทำความเข้าใจลักษณะการบันทึกงาน  พร้อมกับที่ Microsoft Dynamics Lifecycle Services (LCS) และ Microsoft Azure DevOps จะยินยอมให้คุณสร้าง ตั้งค่าคอนฟิก รัน ตรวจสอบ และรายงานเกี่ยวกับการทดสอบ
- เพิ่มประสิทธิภาพผู้ใช้งานเฉพาะทางให้สามารถบันทึกงานของธุรกิจได้โดยใช้ตัวบันทึกงาน จากนั้น แปลงการบันทึกงานลงในชุดของการทดสอบอัตโนมัติโดยไม่ต้องเขียนรหัสต้นทาง

## <a name="before-you-begin"></a>ก่อนที่คุณจะเริ่มต้น

### <a name="prerequisites"></a>ข้อกำหนดเบื้องต้น

- สภาพแวดล้อมที่รัน Microsoft Dynamics 365 for Finance and Operations รุ่น 10.0 (เมษายน 2019) หรือรุ่นที่ใหม่กว่า จำเป็นสำหรับการสอนในบทนี้ สำหรับลูกค้าที่ใช้ Microsoft Dynamics 365 for Finance and Operations Enterprise edition 7.3 การปรับปรุงแพลตฟอร์ม 20 (PU20) หรือใหม่กว่า ยังได้รับการสนับสนุนด้วยเช่นกัน
- ผู้ใช้ต้องมีสิทธิ์ในการดูแลระบบสำหรับสภาพแวดล้อมนี้
- คุณต้องมีสิทธิเข้าถึงผู้เช่าของลูกค้า LCS และ Azure DevOps (ก่อนหน้านี้เรียกว่า Microsoft Visual Studio Team Services \[VSTS\])
- ผู้ใช้ที่กำลังสร้างและจัดการชุดทดสอบต้องมีสิทธิ์การใช้งาน Test Plans หรือ Test Manager ของ Azure DevOps นอกจากนี้ สิทธิ์การใช้งานต่อไปนี้ยังช่วยให้คุณสามารถเข้าถึง Test Plans:
    - ลิขสิทธิ์ Visual Studio Enterprise
    - ลิขสิทธิ์ Visual Studio Test Professional
    - สิทธิ์การใช้งานของสมาชิกของแพลตฟอร์ม MSDN
- RSAT ต้องมีสิทธิ์เข้าถึงสภาพแวดล้อมการทดสอบผ่านทางเว็บเบราเซอร์
- เมื่อต้องการสร้างและแก้ไขพารามิเตอร์การทดสอบ คุณต้องติดตั้ง Microsoft Excel

## <a name="configure-azure-devops"></a>ตั้งค่าคอนฟิก Azure DevOps

### <a name="user-eligibility"></a>การมีสิทธิ์ของผู้ใช้

ตรวจสอบให้แน่ใจว่ามีการสร้างผู้ใช้ใน Azure DevOps และมีระดับการบอกรับเป็นสมาชิกที่ให้สิทธิ์เข้าถึง Azure Test Plans ต้องมีสิทธิ์การใช้งาน Azure DevOps Test Plans เฉพาะเมื่อผู้ใช้จะสร้างและจัดการกรณีการทดสอบ (กล่าวคือ ไม่ใช่ผู้ใช้ RSAT ทั้งหมดต้องมีสิทธิ์การใช้งานนี้) สำหรับข้อมูลเกี่ยวกับข้อกำหนดสิทธิ์การใช้งาน โปรดดู [ข้อกำหนดสิทธิ์การใช้งาน](https://docs.microsoft.com/azure/devops/test/manual-test-permissions#license-requirements)

### <a name="create-a-new-azure-devops-project"></a>สร้างโครงการ Azure DevOps ใหม่
RSAT ใช้ Azure Devops สำหรับกรณีทดสอบและการจัดการชุดทดสอบ การรายงาน และการตรวจสอบผลลัพธ์การรันการทดสอบ 

> [!NOTE]
> คุณสามารถใช้โครงการ Azure DevOps ที่มีอยู่ได้ อย่างไรก็ตาม ถ้ามีการตั้งค่าโครงการ Azure DevOps ที่มีอยู่ของคุณเพื่อให้มีเท็มเพลตที่กำหนดเอง คุณจะได้รับข้อผิดพลาด "ความล้มเหลวในการซิงค์กับ VSTS" เมื่อคุณซิงค์กรณีทดสอบจากตัวทำแบบจำลองกระบวนการทางธุรกิจ (BPM) กับ Azure DevOps (ดูส่วน [ทดสอบการซิงโครไนส์จาก BPM ไปยัง Azure DevOps](#test-the-synchronization-from-bpm-to-azure-devops) ) ถ้ามีการปฏิบัติตามแนวทางที่พึงปฏิบัติต่อไปนี้สำหรับเท็มเพลตที่กำหนดเอง คุณจะสามารถซิงค์กรณีการทดสอบกับ Azure DevOps ได้ (แนวทางที่พึงปฏิบัติเหล่านี้จะแสดงรายการอยู่ในข้อความแสดงข้อผิดพลาด)

- อย่าลบชนิดรายการงานหรือฟิลด์แบบสำเร็จรูปใดๆ
- อย่าลบสถานะของชนิดรายการงานใดๆ
- อย่าเพิ่มฟิลด์ที่จำเป็นใดๆ ไปยังชนิดรายการงาน

![ข้อความแสดงข้อผิดพลาดที่มีรายการของแนวทางที่พึงปฏิบัติ](./media/setup_rsa_tool_02.png)

มิฉะนั้น สำหรับบทช่วยสอนนี้ เราขอแนะนำให้คุณสร้างโครงการ Azure DevOps ใหม่ สำหรับข้อมูลเพิ่มเติม ดู [ปัญหาเมื่อซิงค์กับ BPM โดยใช้เท็มเพลตการประมวลผล Azure DevOps (VSTS) แบบกำหนดเอง](https://blogs.msdn.microsoft.com/lcs/2018/11/28/issues-when-syncing-to-bpm-using-a-custom-azure-devops-vsts-process-template/)

1. เปิด Azure DevOps URL (`https://dev.azure.com/<Azure DevOps Name>`)
2. เลือก **สร้างโครงการ** ที่มุมขวาบนของหน้า Azure DevOps

    ![สร้างปุ่มโครงการ](./media/setup_rsa_tool_03.png)

3. กรอกข้อมูลในฟิลด์ต่อไปนี้ แล้วเลือก **สร้าง**:

    - **ชื่อโครงการ**
    - **การควบคุมรุ่น** – เลือก **Team Foundation Version Control** โปรดทราบว่าตัวเลือกเริ่มต้น **Git** ไม่ได้รับการสนับสนุน
    - **กระบวนการรายการงาน**

    ![กล่องโต้ตอบสร้างโครงการใหม่](./media/setup_rsa_tool_04.png)

### <a name="create-a-personal-access-token"></a>สร้างโทเคนการเข้าถึงส่วนบุคคล

ในบทช่วยสอนนี้ คุณจะใช้ LCS Business Process Modeler (BPM) เพื่อสร้างไลบรารีกรณีทดสอบ และซิงโครไนส์กรณีการทดสอบของคุณกับ Azure DevOps คุณจะต้องใช้โทเคนการเข้าถึงส่วนบุคคลเพื่อซิงค์ BPM กับ Azure DevOps

1. เลือกไอคอนโพรไฟล์ที่มุมด้านขวาบนของหน้าสำหรับโครงการ Azure DevOps ของคุณ แล้วเลือก **ความปลอดภัย**

    ![คำสั่งรักษาความปลอดภัย](./media/setup_rsa_tool_05.png)

2. ในบานหน้าต่างด้านซ้าย ภายใต้ **ความปลอดภัย** เลือก **โทเคนการเข้าถึงส่วนบุคคล** จากนั้น เลือก **โทเคนใหม่**

    ![ปุ่มโทเคนใหม่บนแท็บโทเคนการเข้าถึงส่วนบุคคลในการตั้งค่าผู้ใช้](./media/setup_rsa_tool_06.png)

3. กรอกข้อมูลในฟิลด์ต่อไปนี้ แล้วเลือก **สร้าง**:

    - **ชื่อ**
    - **การหมดอายุ (UTC)** – เลือก **กำหนดการกำหนดเองแล้ว** และจากนั้น ใช้ตัวเลือกวันที่เพื่อเลือกวันที่พร้อมใช้งานล่าสุด
    - **ขอบเขต** – เลือกตัวเลือก **การเข้าถึงแบบเต็ม**

    ![กล่องโต้ตอบสร้างโทเคนการเข้าถึงส่วนบุคคลใหม่](./media/setup_rsa_tool_07.png)

    > [!NOTE]
    > จดบันทึกโทเคนการเข้าถึงส่วนบุคคลที่สร้างขึ้น คุณจะต้องใช้ในภายหลัง เมื่อคุณตั้งค่าการตั้งค่าคอนฟิก RSAT

    ![โทเคนการเข้าถึงส่วนบุคคล](./media/setup_rsa_tool_08.png)

## <a name="configure-the-lcs-project"></a>ตั้งค่าคอนฟิกโครงการ LCS

คุณต้องมีโครงการ Lifecycle Services (LCS) สำหรับไลบรารีการทดสอบหลักของคุณ LCS Business Process Modeler (BPM) ถูกใช้เป็นไลบรารีหลักสำหรับกรณีทดสอบของคุณ BPM ใช้ในการจัดการและเผยแพร่ไลบรารีการทดสอบในโครงการ LCS ตัวอย่างเช่น คู่ค้า Microsoft หรือผู้จัดจำหน่ายซอฟต์แวร์อิสระ (ISV) ซึ่งสร้างไลบรารีการทดสอบ จะนำกรณีทดสอบออกใช้ในรูปแบบของไลบรารี BPM ใน BPM กรณีทดสอบจะถูกจัดระเบียบตามกระบวนการทางธุรกิจ BPM ไม่ได้กำหนดลำดับการดำเนินการหรือความถี่ของการผ่านการทดสอบ รายละเอียดเหล่านี้จะถูกจัดการใน Azure DevOps ตามที่อธิบายไว้ในภายหลังในหัวข้อนี้  

สำหรับโครงการ LCS ของคุณ คุณสามารถใช้การดำเนินงานของลูกค้าที่มีอยู่ หรือโครงการคู่ค้า

### <a name="update-the-lcs-language"></a>ปรับปรุงภาษา LCS

> [!NOTE]
> สำหรับอินเทอร์เฟสผู้ใช้ (UI) เพื่อแสดงกระบวนการทางธุรกิจอย่างถูกต้อง ต้องตั้งค่าภาษาที่ต้องการ LCS เป็น **ภาษาอังกฤษ (สหรัฐอเมริกา)**

1. ไปที่โครงการสำหรับการใช้งาน LCS
2. เลือกปุ่ม **การตั้งค่า** (สัญลักษณ์เกียร์) ในมุมขวาบน และจากนั้น เลือก **การกำหนดลักษณะภาษา**

    ![คำสั่งบนเมนูการตั้งค่า](./media/setup_rsa_tool_09.png)

3. ในฟิลด์ **การกำหนดลักษณะภาษา** เลือก **ภาษาอังกฤษ (สหรัฐอเมริกา)** แล้วเลือก **บันทึก**

    ![แท็บการกำหนดลักษณะภาษาในการตั้งค่าผู้ใช้](./media/setup_rsa_tool_10.png)

### <a name="configure-lcs-to-connect-to-the-azure-devops-project"></a>ตั้งค่าคอนฟิก LCS เพื่อเชื่อมต่อกับโครงการ Azure DevOps

ถ้าคุณสร้างโครงการ Azure DevOps ใหม่ก่อนหน้านี้ ให้ตั้งค่าคอนฟิกโครงการ LCS เพื่อเชื่อมต่อ มิฉะนั้น ถ้าโครงการ LCS ของคุณเชื่อมต่อกับโครงการ Azure DevOps ของคุณอยู่แล้ว คุณสามารถดำเนินการต่อไปยังส่วนถัดไปได้

1. ไปที่โครงการสำหรับการใช้งาน LCS
2. เลือกปุ่ม **เมนู** แล้วจากนั้น เลือก **การตั้งค่าโครงการ**

    ![คำสั่งการตั้งค่าโครงการ](./media/setup_rsa_tool_11.png)

3. ในบานหน้าต่างด้านซ้าย ให้เลือก **Visual Studio Team Services** แล้วจากนั้น เลือก **ตั้งค่า Visual Studio Team Services**

    ![แท็บ Visual Studio Team Services ในการตั้งค่าโครงการ](./media/setup_rsa_tool_12.png)

4. ในฟิลด์ **URL ของไซต์ Azure DevOps** ให้ป้อน URL ของไซต์ Azure DevOps ในฟิลด์ **โทเคนการเข้าถึงส่วนบุคคล** ให้ป้อนโทเคนการเข้าถึงส่วนบุคคลที่สร้างไว้ก่อนหน้านี้

    > [!NOTE]
    > ถึงแม้ว่าขณะนี้ VSTS ถูกเรียกว่า Azure DevOps เพื่อเชื่อมต่อ LCS กับโครงการ Azure DevOps ของคุณ ให้ใช้ URL เก่า ตัวอย่างเช่น URL ของ Azure DevOps ที่ใช้ในบทช่วยสอนนี้คือ `https://dev.azure.com/D365FOFastTrack/` อย่างไรก็ตาม ในภาพประกอบต่อไปนี้ จะมีการป้อนเป็น `https://D365FOFastTrack.visualstudio.com/`

    ![ขั้นตอนที่ 1 ตั้งค่า Visual Studio Team Services](./media/setup_rsa_tool_13.png)

5. เลือก **ดำเนินการต่อ**
6. ในฟิลด์ **โครงการ Visual Studio Team Services** เลือกโครงการ VSTS ในไซต์ที่เลือกเพื่อเชื่อมโยงกับโครงการ LCS ฟิลด์ **เท็มเพลตการประมวลผล** ถูกตั้งค่าเป็น **Agile** โดยค่าเริ่มต้น สำหรับเท็มเพลตที่กำหนดเอง ให้ทบทวนแนวทางปฏิบัติที่ดีที่สุดในส่วน [สร้างโครงการ Azure DevOps ใหม่](#create-a-new-azure-devops-project) จากนั้น เลือก **ดำเนินการต่อ**

    ![ขั้นตอนที่ 2 ใน ตั้งค่า Visual Studio Team Services](./media/setup_rsa_tool_14.png)

7. ตรวจทานการตั้งค่าของคุณ แล้วเลือก **บันทึก**

    ![ขั้นตอนที่ 3 ใน ตั้งค่า Visual Studio Team Services](./media/setup_rsa_tool_15.png)

8. เลือก **อนุญาต** เพื่ออนุมัติให้ LCS สามารถเข้าถึงไซต์ Azure DevOps ที่ตั้งค่าคอนฟิกในนามของคุณ และเพื่อเปิดใช้งานคุณลักษณะที่รวมกับ VSTS

    ![อนุมัติปุ่ม](./media/setup_rsa_tool_16.png)

9. กล่องข้อความปรากฏขึ้น ซึ่งกล่าวว่า "คุณกำลังจะเปลี่ยนเส้นทางไปยังไซต์ภายนอกเพื่ออนุมัติให้ LCS เชื่อมต่อกับ Visual Studio Team Services ในนามของคุณ ดำเนินการหรือไม่?" เลือก **ใช่**

    ![กล่องข้อความ](./media/setup_rsa_tool_17.png)

10. เลือก **ยอมรับ**

    ![การอนุมัติการเข้าถึง](./media/setup_rsa_tool_18.png)

> [!NOTE]

## <a name="install-and-configure-rsat"></a>ติดตั้งและตั้งค่าคอนฟิก RSAT

RSAT สามารถติดตั้งบนคอมพิวเตอร์เครื่องใดก็ได้ที่รัน Windows 10 และสามารถเชื่อมต่อกับสภาพแวดล้อมผ่านเว็บเบราเซอร์ (Internet Explorer หรือ Google Chrome)

> [!NOTE]
> ก่อนที่คุณจะติดตั้งเครื่องมือรุ่นใหม่ เราขอแนะนำให้คุณถอนการติดตั้งรุ่นก่อนหน้านี้

### <a name="install-an-authentication-certificate"></a>ติดตั้งใบรับรองการพิสูจน์ตัวจริง

เมื่อต้องการเปิดใช้งานการรับรองความถูกต้อง คุณต้องสร้างและติดตั้งใบรับรองบนคอมพิวเตอร์เครื่องเดียวกันที่ RSAT กำลังรันอยู่

#### <a name="generate-a-certificate"></a>สร้างใบรับรอง

1. เมื่อต้องการสร้างไฟล์ใบรับรอง ให้เปิดหน้าต่าง Microsoft Windows PowerShell ในฐานะผู้ดูแลระบบ และรันคำสั่งต่อไปนี้

    ```
    $certificate = New-SelfSignedCertificate -dnsname 127.0.0.1 -CertStoreLocation cert:\LocalMachine\My -FriendlyName "D365 Automated testing certificate" -Provider "Microsoft Strong Cryptographic Provider"
    ```

2. เปิดโปรแกรมจัดการใบรับรองโดยป้อน **certlm.msc** ในกล่องโต้ตอบ **รัน** และยืนยันว่าใบรับรอง **D365 ใบรับรองการทดสอบแบบอัตโนมัติ** ถูกสร้างภายใต้ **ส่วนบุคคล \> ใบรับรอง**

    > [!NOTE]
    > ตรวจสอบให้แน่ใจว่าได้ป้อน **certlm.msc** ไม่ใช่ **certmgr.msc** เนื่องจากใบรับรองถูกจัดเก็บอยู่บนคอมพิวเตอร์เฉพาะที่

    ![D365 ใบรับรองการทดสอบแบบอัตโนมัติ](./media/setup_rsa_tool_46.png)

3. คลิกขวาที่ใบรับรอง แล้วจากนั้น เลือก **คัดลอก**
4. ไปที่ **หน่วยงานที่ออกใบรับรองหลักที่เชื่อถือได้ \> ใบรับรอง**

    ![โฟลเดอร์ใบรับรองภายใต้โฟลเดอร์หน่วยงานที่ออกใบรับรองหลักที่เชื่อถือได้](./media/setup_rsa_tool_47.png)

5. ในเมนู **การดำเนินการ** ให้เลือก **วาง** เพื่อคัดลอกใบรับรองไปยังสถานที่ **หน่วยงานที่ออกใบรับรองหลักที่เชื่อถือได้**

    ![คำสั่งวางบนเมนูการดำเนินการ](./media/setup_rsa_tool_48.png)

6. เมื่อต้องการดูรหัสประจำตัวของใบรับรองที่ติดตั้งไว้ แต่ไม่มีช่องว่างหรืออักขระพิเศษ ให้เปิดหน้าต่าง Windows PowerShell ในฐานะผู้ดูแลระบบ และรันคำสั่งต่อไปนี้

    ```
    cd Cert:\LocalMachine\My

    Get-ChildItem | Where-Object { $_.Subject -like "CN=127.0.0.1" }
    ```

    > [!NOTE]
    > บันทึกรหัสประจำตัว เนื่องจากคุณจะต้องใช้ในภายหลัง เมื่อคุณปรับปรุงไฟล์ .wif สำหรับ Application Object Server (AOS) และตั้งค่าการตั้งค่าคอนฟิก RSAT

#### <a name="configure-the-aos-computer-to-trust-the-connection"></a>ตั้งค่าคอนฟิกคอมพิวเตอร์ AOS เพื่อให้ความเชื่อถือในการเชื่อมต่อ

> [!NOTE]
> หากสภาพแวดล้อมเป็นสภาพแวดล้อมระดับ 2+ รหัสประจำตัวของใบรับรองต้องได้รับการอัพเดตในไฟล์ wif.config สำหรับคอมพิวเตอร์ AOS **ทั้งหมด** สำหรับสภาพแวดล้อมนั้น ๆ

1. สร้างการเชื่อมต่อ Remote Desktop Protocol (RDP) ไปยังคอมพิวเตอร์ AOS รายละเอียดการลงชื่อเข้าใช้จะพร้อมใช้งานในหน้ารายละเอียดของสภาพแวดล้อมใน LCS
2. เปิด Microsoft Internet Information Services (IIS) และค้นหา **AOSService** ในรายการของไซต์

    ![AOSService ในรายการของไซต์](./media/setup_rsa_tool_49.png)

3. คลิกขวา **สำรวจ** เพื่อเปิดโฟลเดอร์ **\<ไดรฟ์\>: \\AosService\\WebRoot** ค้นหาไฟล์ **wif.config**

    ![ไฟล์ Wif.config ในโฟลเดอร์ WebRoot](./media/setup_rsa_tool_50.png)

4. ปรับปรุงไฟล์ **wif.config** โดยการเพิ่มรายการหน่วยงานใหม่สำหรับชื่อใบรับรองและชื่อหน่วยงานของคุณ ดังที่แสดงในตัวอย่างต่อไปนี้

    ```
    <issuerNameRegistry type="Microsoft.Dynamics.AX.Security.SharedUtility.AxIssuerNameRegistry,Microsoft.Dynamics.AX.Security.SharedUtility">
        <authority name="CN=127.0.0.1">
            <keys>
                <add thumbprint="xxxxxxxxxxxxxxxxxxxx" />
                <add thumbprint="xxxxxxxxxxxxxxxxxxxx" />
            </keys>
            <validIssuers>
                <add name="CN=127.0.0.1" />
            </validIssuers>
        </authority>
    ```

    > [!NOTE]
    > ถ้าผู้ใช้หลายรายใช้แอพลิเคชันเดียวกัน ผู้ใช้แต่ละรายจะต้องสร้างรหัสประจำตัวที่แยกต่างหาก และต้องเพิ่มรหัสประจำตัวแต่ละรหัสลงในส่วนของ **\<คีย์\>**

5. ถ้ามีคอมพิวเตอร์ AOS มากกว่าหนึ่งเครื่อง ให้ทำซ้ำขั้นตอนที่ 3 ถึง 4 สำหรับคอมพิวเตอร์เพิ่มเติมแต่ละเครื่อง

    > [!NOTE]
    > แตกต่างจากสภาพแวดล้อมระดับที่ 2 สภาพแวดล้อมระดับที่ 2 ใหม่จะถูกปรับใช้กับอินสแตนซ์ AOS สองรายการ

6. รัน **iisreset** บนคอมพิวเตอร์ AOS ทั้งหมด

    > [!NOTE]
    > ถ้าคุณไม่มีสิทธิ์ของผู้ดูแลระบบในการรัน **iisreset** บนคอมพิวเตอร์ระดับที่ 1 บนหน้ารายละเอียดสภาพแวดล้อมใน LCS ให้เลือกรักษา > เริ่มต้นบริการอีกครั้ง

#### <a name="tier-2-environment"></a>สภาพแวดล้อมระดับ 2+

ถ้ามีการใช้สภาพแวดล้อมระดับ 2+ (มาตรฐานการยอมรับของการทดสอบ Sandbox หรือสูงกว่า) ให้ยืนยันหรือตั้งค่าการลงทะเบียนต่อไปนี้บนคอมพิวเตอร์ที่มีการติดตั้ง RSAT ขั้นตอนนี้จำเป็น เนื่องจากจะช่วยให้คุณสามารถหลีกเลี่ยงข้อผิดพลาดในการตรวจสอบความถูกต้องหรือการเชื่อมต่อ RSAT

```
if ((Test-Path HKLM:\SOFTWARE\Wow6432Node\Microsoft\.NETFramework\v4.0.30319))
{
    Set-ItemProperty HKLM:\SOFTWARE\Wow6432Node\Microsoft\.NETFramework\v4.0.30319 -Name SchUseStrongCrypto -Value 1 -Type dword -Force -Confirm:$false
}
```

### <a name="install-rsat"></a>ติดตั้ง RSAT

1. ไปที่ <https://www.microsoft.com/download/details.aspx?id=57357> แล้วเลือก **ดาวน์โหลด**
2. เลือกไฟล์ทั้งหมด แล้วจากนั้น เลือก **ถัดไป**

3. คลิกสองครั้งที่แพคเกจ .msi เพื่อรันตัวติดตั้ง จากนั้น เมื่อการติดตั้งเสร็จสมบูรณ์ ให้เลือก **เสร็จสิ้น**

    ![ไฟล์ตัวติดตั้ง RSAT](./media/setup_rsa_tool_52.png)

### <a name="install-selenium-and-browser-drivers"></a>ติดตั้ง Selenium และไดรเวอร์ของเบราว์เซอร์

ในรุ่นก่อนหน้าของ RSAT คุณต้องติดตั้งไดรเวอร์ของ Selenium และเบราว์เซอร์ คุณไม่จำเป็นต้องติดตั้งไดรเวอร์เหล่านี้อีกต่อไป เนื่องจากมีการติดตั้งโดยอัตโนมัติ

1. ในครั้งแรกที่คุณเปิด RSAT คุณจะได้รับพร้อมท์ให้ดาวน์โหลดและติดตั้ง Selenium โดยอัตโนมัติ สำหรับข้อมูลเพิ่มเติม ให้ดูที่ส่วน [ตั้งค่าคอนฟิก RSAT](#configure-rsat)
2. ก่อนที่คุณจะสามารถรันกรณีการทดสอบ คุณจะได้รับพร้อมท์ให้ดาวน์โหลดและติดตั้งไดรเวอร์ของเบราว์เซอร์ที่สอดคล้องกับเบราว์เซอร์เริ่มต้นที่เลือกไว้ในการตั้งค่าคอนฟิก RSAT โดยอัตโนมัติ สำหรับข้อมูลเพิ่มเติม ให้ดูที่ส่วน [โหลดและรันกรณีการทดสอบ](#load-and-run-test-cases)

    ![กล่องโต้ตอบแผนการทดสอบใหม่](./media/setup_rsa_tool_55.png)

4. เลือกเครื่องหมายบวก (**+**) แล้วจากนั้น เลือก **ชุดแบบคงที่** เพื่อสร้างชุดแบบคงที่ภายใต้แผนการทดสอบใหม่ ตั้งชื่อชุดการทดสอบใหม่ **T01 – การผลิตตามสินค้าคงคลัง**

    ![การสร้างชุดแบบคงที่](./media/setup_rsa_tool_56.png)

    ![กรณีการทดสอบที่เพิ่มในชุดการทดสอบ](./media/setup_rsa_tool_59.png)

### <a name="configure-rsat"></a>ตั้งค่าคอนฟิก RSAT

1. เปิด RSAT

    ![ไอคอน RSAT](./media/setup_rsa_tool_60.png)

2. คุณได้รับข้อความแจ้งเตือนที่ระบุว่า "Regression Suite Automation Tool ต้องมี Selenium คุณต้องการดาวน์โหลดและติดตั้งโดยอัตโนมัติเดี๋ยวนี้หรือไม่?" เลือก **ใช่**

    ![ข้อความเตือน](./media/setup_rsa_tool_61.png)

3. เลือกปุ่ม **การตั้งค่า** (สัญลักษณ์รูปเกียร์) ที่มุมขวาบน และจากนั้น ในกล่องโต้ตอบที่ปรากฏขึ้น ให้กรอกข้อมูลในฟิลด์ต่อไปนี้:

    - **Url ของ Azure DevOps** – ป้อน URL ของโครงการ Azure DevOps ของคุณ เช่น `https://yourAzureDevOpsUrlHere.visualStudio.com`
    - **โทเคนการเข้าถึง** – ป้อนโทเคนการเข้าถึงซึ่งช่วยให้เครื่องมือสามารถเชื่อมต่อกับ Azure DevOps ใช้โทเคนการเข้าถึงส่วนบุคคลที่คุณสร้างไว้ก่อนหน้านี้ในบทช่วยสอนนี้ สำหรับข้อมูลเพิ่มเติม ดู [ตรวจสอบสิทธิ์การเข้าถึงโดยใช้โทเคนการเข้าถึงส่วนบุคคล](https://www.visualstudio.com/docs/setup-admin/team-services/use-personal-access-tokens-to-authenticate)
    - **ชื่อโครงการ** – เลือกชื่อของโครงการ Azure DevOps ของคุณ
    - **แผนการทดสอบ** – เลือกแผนการทดสอบ Azure DevOps ที่มีกรณีการทดสอบของคุณ สำหรับข้อมูลเพิ่มเติม ดู [สร้างแผนการทดสอบและชุดการทดสอบ](https://www.visualstudio.com/docs/test/manual-exploratory-testing/getting-started/create-a-test-plan) หลังจากที่คุณเลือกแผนการทดสอบ ให้เลือก **ทดสอบการเชื่อมต่อ** เพื่อทดสอบการเชื่อมต่อของคุณกับ Azure DevOps
    - **Hostname** – ป้อนชื่อโฮสต์ของสภาพแวดล้อมที่ทดสอบ เช่น **\<myaos\>.cloudax.dynamics.com** อย่ารวมคำนำหน้า **https://** หรือ **http://**
    - **SOAP Hostname** – ป้อนชื่อโฮสต์ SOAP ของสภาพแวดล้อมที่ทดสอบ โดยทั่วไป ชื่อโฮสต์ SOAP จะเหมือนกับชื่อโฮสต์ แต่มีคำต่อท้าย **soap** นี่คือตัวอย่าง **\<myaos\>soap.cloudax.dynamics.com** อย่ารวมคำนำหน้า **https://** หรือ **http://**

        > [!NOTE]
        > เมื่อต้องการค้นหาชื่อโฮสต์และชื่อโฮสต์ SOAP ให้เปิดโปรแกรมจัดการ IIS คลิกขวา **ไซต์ \> AOSService** และจากนั้น เลือก **แก้ไขการผูก** ค่าในคอลัมน์ **ชื่อโฮสต์** จะให้ชื่อโฮสต์และชื่อโฮสต์ SOAP (ชื่อโฮสต์ SOAP มีคำต่อท้าย **soap** ใน URL)

        ![ชื่อโฮสต์และชื่อโฮสต์ SOAP ในคอลัมน์ชื่อโฮสต์](./media/setup_rsa_tool_63.png)

    - **ชื่อผู้ใช้ที่ดูแลระบบ** – ป้อนที่อยู่อีเมล์ของผู้ใช้ที่ดูแลระบบในสภาพแวดล้อมที่ทดสอบ
    - **รหัสประจำตัว** – ป้อนรหัสประจำตัวของใบรับรองการพิสูจน์ตัวจริง ดังที่อธิบายไว้ก่อนหน้านี้ในบทช่วยสอนนี้
    - **ไดเรกทอรีการทำงาน** – ระบุตำแหน่งโฟลเดอร์สำหรับจัดเก็บไฟล์การทำงานอัตโนมัติของการทดสอบ เช่น ไฟล์ข้อมูลการทดสอบของ Excel ตัวอย่างเช่น ป้อน หรือเลือก **C:\\Temp\\RegressionTool**

        > [!NOTE]
        > ถ้าชื่อของโฟลเดอร์มีช่องว่าง การดำเนินการจะล้มเหลวเนื่องจากไม่พบโฟลเดอร์ ปัญหานี้เป็นปัญหาที่ทราบ และควรได้รับการแก้ไขในเครื่องมือรุ่นล่าสุด

    - **เบราเซอร์เริ่มต้น** – เลือก **Internet Explorer** หรือ **Google Chrome** อย่างใดอย่างหนึ่ง ตรวจสอบให้แน่ใจว่าได้ติดตั้งไดรเวอร์ของเบราว์เซอร์ที่เหมาะสมแล้ว
    - **การหมดเวลาของการรันการทดสอบ** – ระบุรอบระยะเวลาการหมดเวลาเป็นนาทีสำหรับการรันการทดสอบ เมื่อรอบระยะเวลาของการหมดเวลาผ่านไป windows ที่ใช้งานอยู่ทั้งหมดจะถูกปิด และกรณีทดสอบที่ค้างอยู่จะล้มเหลว
    - **การหมดเวลาของการดำเนินการทดสอบ** – ฟิลด์นี้จะควบคุมรอบระยะเวลาการหมดเวลาเป็นนาที สำหรับการร้องขอเซิร์ฟเวอร์ของสภาพแวดล้อม Finance and Operation โดยปกติแล้ว ค่าเริ่มต้น (2 นาที) ควรจะเพียงพอ อย่างไรก็ตาม สำหรับสภาพแวดล้อมที่ช้าลง คุณอาจต้องการเพิ่มค่า ถ้าข้อผิดพลาดที่เกี่ยวข้องกับการหมดเวลาเกิดขึ้น
    - **ชื่อบริษัท** – ป้อนชื่อบริษัทที่จะใช้เป็นบริษัทเริ่มต้นของคุณ เมื่อมีการสร้างไฟล์พารามิเตอร์ Excel คุณสามารถเปลี่ยนบริษัทได้ในภายหลัง โดยแก้ไขไฟล์พารามิเตอร์ Excel

    ![กล่องโต้ตอบการตั้งค่า](./media/setup_rsa_tool_62.png)

4. เลือก **นำไปใช้** เพื่อนำไปใช้และบันทึกการตั้งค่าของคุณ

    เมื่อต้องการบันทึกการตั้งค่าปัจจุบันของคุณลงในไฟล์การตั้งค่าคอนฟิกบนคอมพิวเตอร์ของคุณ ให้เลือก **บันทึกเป็น** เมื่อต้องการคืนค่าการตั้งค่าของคุณจากไฟล์การตั้งค่าคอนฟิกบนคอมพิวเตอร์ของคุณ ให้เลือก **เปิด**

5. เลือก **ปิด** เพื่อปิดกล่องโต้ตอบ

### <a name="load-and-run-test-cases"></a>โหลดและรันกรณีการทดสอบ

1. เลือก **โหลด** เพื่อโหลดแผนการทดสอบ **แผนการทดสอบ RSAT** จากโครงการ Azure DevOps

    ![ปุ่มโหลด](./media/setup_rsa_tool_64.png)

2. เลือกกรณีการทดสอบ **สร้างผลิตภัณฑ์ใหม่** จากชุดการทดสอบ แล้วเลือก **สร้าง \> สร้างไฟล์การดำเนินการทดสอบและไฟล์พารามิเตอร์**

    ![คำสั่งสร้างไฟล์การดำเนินการทดสอบและไฟล์พารามิเตอร์บนเมนูสร้าง](./media/setup_rsa_tool_65.png)

    ด้วยวิธีนี้ คุณสามารถเลือก **โหลด** เพื่อโหลดไฟล์พารามิเตอร์ (และไฟล์ระบบอัตโนมัติ) จากกรณีการทดสอบโดยตรงจาก Azure DevOps คุณไม่จำเป็นต้องสร้างไฟล์พารามิเตอร์ใหม่อีกครั้ง วิธีการนี้จะกลายเป็นเรื่องสำคัญในภายหลัง เมื่อคุณต้องการรักษาการปรับเปลี่ยนในไฟล์พารามิเตอร์ และไม่ต้องการให้มีการเขียนทับ

    - **.dll** – คอมไพล์ไฟล์ระบบอัตโนมัติเป็นแอสเซมบลี
    - **.xlsx** – ไฟล์พารามิเตอร์ Excel
    - **.xml** – การบันทึกไฟล์

### <a name="modify-the-excel-parameter-file"></a>ปรับเปลี่ยนไฟล์พารามิเตอร์ของ Excel

1. เปิด RSAT
2. เลือกกล่องการทดสอบ แล้วจากนั้น เลือก **แก้ไข** เพื่อเปิดไฟล์พารามิเตอร์ Excel

    บนแผ่นงาน **EcoResProductCreate** โปรดทราบว่าค่าของฟิลด์ **หมายเลขผลิตภัณฑ์** มีรหัสแบบตายตัว คุณต้องปรับปรุงฟิลด์นี้เป็นหมายเลขผลิตภัณฑ์ใหม่ ก่อนที่คุณจะรันกรณีการทดสอบอีกครั้ง

3. เมื่อต้องการสร้างหมายเลขผลิตภัณฑ์ที่ไม่ซ้ำกันสำหรับการรันแต่ละครั้งโดยไม่ต้องเปิดไฟล์พารามิเตอร์ Excel อีกครั้ง และปรับปรุงหมายเลขผลิตภัณฑ์ด้วยตนเองทุกครั้ง ใช้สูตรของ Excel ต่อไปนี้

    ```
    ="RSAT_"&TEXT(NOW(),"yyymmddhhmm")
    ```

    > [!NOTE]
    > นอกจากแท็บ **ทั่วไป** ไฟล์พารามิเตอร์ของ Excel จะมีแท็บข้อมูลเพื่อให้สามารถเข้าชมแบบฟอร์มของการทดสอบได้ทุกหน้า

    ![ฟิลด์หมายเลขผลิตภัณฑ์](./media/setup_rsa_tool_81.png)

    ![ข้อความการอัปโหลดสำเร็จ](./media/setup_rsa_tool_82.png)

    > [!NOTE]
    > เมื่อต้องการรันกรณีการทดสอบในบริบทของผู้ใช้ที่ระบุ ให้ป้อนรหัสอีเมลของผู้ใช้ในฟิลด์ **ผู้ใช้การทดสอบ** บนแท็บ **ทั่วไป** ของไฟล์พารามิเตอร์ Excel ในรุ่นล่าสุดของ RSAT จะมีการปรับปรุงโครงร่างของฟิลด์ในไฟล์พารามิเตอร์ของ Excel แต่แนวคิดยังคงอยู่เหมือนเดิม

### <a name="chaining-of-test-cases"></a>กลุ่มของกรณีการทดสอบ

คุณลักษณะที่สำคัญอย่างหนึ่งของ RSAT คือการเชื่อมโยงของกรณีการทดสอบ (นั่นคือ ความสามารถของการทดสอบเพื่อส่งผ่านค่าไปยังการทดสอบอื่นๆ) กรณีการทดสอบจะรันตามลำดับที่กำหนดในแผนการทดสอบ Azure DevOps (ลำดับนี้ยังสามารถถูกปรับปรุงได้ในเครื่องมือการทดสอบเอง) ดังนั้น ถ้าคุณต้องการส่งผ่านตัวแปรจากกรณีการทดสอบหนึ่งไปยังอีกกรณีหนึ่ง อาจเป็นเรื่องสำคัญมากที่การทดสอบจะต้องอยู่ในลำดับที่ถูกต้อง

ในส่วนนี้ คุณจะสร้างตัวแปรที่บันทึกไว้ในกรณีการทดสอบครั้งแรก สร้างกรณีการทดสอบที่สอง และส่งผ่านตัวแปรที่บันทึกไว้จากกรณีการทดสอบครั้งแรกไปยังกรณีการทดสอบครั้งที่สอง จากนั้น คุณจะสามารถรันกรณีการทดสอบเป็นกรณีการทดสอบที่เชื่อมโยงได้ใน RSAT

    ![เพิ่มขั้นตอนใหม่แล้ว](./media/setup_rsa_tool_93.png)

12. เลือก **ทำการแก้ไขเสร็จสิ้นแล้ว**
13. เลือก **บันทึกไปยัง Lifecycle Services** และเชื่อมโยงการบันทึกงานใหม่กับไลบรารี BPM เดียวกัน และกระบวนการทางธุรกิจที่เกี่ยวข้องกับการบันทึกงานดั้งเดิม สำหรับข้อมูลเพิ่มเติม ดูส่วน [สร้างการบันทึกงาน และบันทึกไปยังไลบรารี BPM](#create-a-task-recording-and-save-it-to-the-bpm-library)
14. ไปยังไลบรารี BPM และเลือก **ซิงค์กรณีการทดสอบ** เพื่อเขียนทับการบันทึกงานที่แนบอยู่กับกรณีการทดสอบใน Azure DevOps ตามที่อธิบายไว้ในส่วน [ทดสอบการซิงโครไนส์จาก BPM ไปยัง Azure DevOps](#test-the-synchronization-from-bpm-to-azure-devops)
15. เปิด RSAT และเลือก **โหลด** เพื่อโหลดกรณีการทดสอบทั้งหมดในชุดทดสอบอีกครั้ง คุณต้องสร้างไฟล์ระบบอัตโนมัติและไฟล์พารามิเตอร์อีกครั้งสำหรับกรณีการทดสอบที่เหมาะสม โดยเลือกกรณีการทดสอบ และจากนั้น เลือก **สร้าง \> สร้างไฟล์การดำเนินการและและไฟล์พารามิเตอร์** ตามที่อธิบายไว้ในส่วน [โหลดและรันกรณีการทดสอบ](#load-and-run-test-cases)

    > [!NOTE]
    > ถ้าไฟล์พารามิเตอร์ Excel ถูกเปิดค้างไว้ การสร้างใหม่จะล้มเหลว ดังนั้น โปรดตรวจสอบให้แน่ใจว่ามีการปิดไฟล์พารามิเตอร์ Excel สำหรับกรณีการทดสอบ ก่อนที่คุณจะสร้างไฟล์พารามิเตอร์ Excel ใหม่

16. เลือก **แก้ไข** เพื่อเปิดไฟล์พารามิเตอร์ Excel ใหม่ คุณจะเห็นรายการ **ตัวแปรที่บันทึกไว้** ใหม่บนบรรทัดที่ 9 ตัวแปรนี้ **{{EcoResProductCreate\_รหัสประจำตัว\_ProductNumber\_คัดลอก}}** ถูกบันทึกไว้ในไฟล์ XML ของการบันทึกงาน และสามารถใช้ในการทดสอบในเวลาต่อมา

    ![รายการตัวแปรที่บันทึกไว้](./media/setup_rsa_tool_94.png)

#### <a name="create-a-new-test-case"></a>สร้างกรณีการทดสอบใหม่

1. ไปที่ไลบรารี BPM ของ **RSAT**
2. เลือกกระบวนการ **กระบวนการทางธุรกิจที่สนับสนุนของตัวอย่าง** และจากนั้น ทางด้านขวา ให้เลือก **โหมดแก้ไข**
3. เปลี่ยนค่าของทั้งฟิลด์ **ชื่อ** และฟิลด์ **คำอธิบาย** เพื่อ **นำผลิตภัณฑ์ออกใช้** จากนั้น เลือก **บันทึก**

    ![ชื่อและคำอธิบายที่เปลี่ยนไปเพื่อนำผลิตภัณฑ์ออกใช้](./media/setup_rsa_tool_95.png)

#### <a name="create-a-new-task-recording-that-has-a-validate-function"></a>สร้างการบันทึกงานใหม่ที่มีฟังก์ชันการตรวจสอบความถูกต้อง

- สร้างการบันทึกงานเพื่อนำผลิตภัณฑ์ที่สร้างไว้ก่อนหน้านี้ออกใช้ไปยังนิติบุคคล USRT สำหรับข้อมูลเพิ่มเติม ดูส่วน [สร้างการบันทึกงาน และบันทึกไปยังไลบรารี BPM](#create-a-task-recording-and-save-it-to-the-bpm-library)

    > [!NOTE]
    > สำหรับกรณีการทดสอบที่เชื่อมโยง เราขอแนะนำให้คุณค้นหาหรือกรองข้อมูลสำหรับเรกคอร์ดที่คุณต้องการด้วย *การพิมพ์ค่าของฟิลด์ด้วยตนเอง* ในลักษณะนี้ เครื่องมือนี้จะสามารถกำหนดว่าจะมีการดำเนินการกับเรกคอร์ดที่ต้องใช้ในกรณีการทดสอบในเวลาต่อมา

    ![การบันทึกงานใหม่ที่มีฟังก์ชันการตรวจสอบความถูกต้อง](./media/setup_rsa_tool_96.png)

    เมื่อภาพประกอบก่อนหน้านี้แสดงขึ้น หลังจากที่พบผลิตภัณฑ์โดยใช้ตัวกรองด่วน แต่ก่อนที่คุณจะเลือก **นำผลิตภัณฑ์ออกใช้** คุณจะตรวจสอบความถูกต้องของค่าของฟิลด์ **หมายเลขผลิตภัณฑ์** เพื่อให้แน่ใจว่ารหัสผลิตภัณฑ์เป็นรหัสผลิตภัณฑ์ที่สร้างไว้ก่อนหน้านี้ เมื่อต้องการตรวจสอบความถูกต้องของค่า คลิกขวาที่ฟิลด์ **หมายเลขผลิตภัณฑ์** และจากนั้น เลือก **ตัวบันทึกงาน \> ตรวจสอบความถูกต้อง \> ค่าปัจจุบัน**

    ![การตรวจสอบความถูกต้องของค่าปัจจุบัน](./media/setup_rsa_tool_97.png)

#### <a name="save-the-task-recording-to-bpm"></a>บันทึกการบันทึกงานไปยัง BPM

1. หลังจากเสร็จสิ้นการบันทึกงาน เลือก **บันทึกไปยัง Lifecycle Services**

    ![ตัวเลือกบันทึก](./media/setup_rsa_tool_98.png)

2. ข้อมูลไลบรารีถูกโหลดจาก LCS

    ![ตัวบ่งชี้ความก้าวหน้า](./media/setup_rsa_tool_99.png)

3. เลือกไลบรารี BPM เพื่อเชื่อมโยงกับการบันทึกงาน สำหรับบทช่วยสอนนี้ ให้เลือกไลบรารี BPM ของ **RSAT** ที่สร้างไว้ก่อนหน้านี้ และ **นำผลิตภัณฑ์ออกใช้** ที่กระบวนการทางธุรกิจอยู่ภายใต้ จากนั้น เลือก **ตกลง**

#### <a name="sync-bpm-to-azure-devops"></a>ซิงค์ BPM กับ Azure DevOps

1. ไปยังไลบรารี BPM และเปิดไลบรารี **RSAT**
2. เลือก **ซิงค์ VSTS** แล้วจากนั้น **ซิงค์กรณีการทดสอบ** สำหรับข้อมูลเพิ่มเติม ดูส่วน [ทดสอบการซิงโครไนส์จาก BPM ไปยัง Azure DevOps](#test-the-synchronization-from-bpm-to-azure-devops)

    หลังจากการซิงโครไนส์เสร็จสมบูรณ์แล้ว รายการงานใหม่และกรณีการทดสอบที่เกี่ยวข้องสำหรับกระบวนการทางธุรกิจ **นำผลิตภัณฑ์ออกใช้** ใน Azure DevOps ที่ **บอร์ด \> รายการงาน**

#### <a name="add-the-new-test-case-to-the-existing-test-suite"></a>เพิ่มกรณีการทดสอบใหม่ไปยังชุดการทดสอบที่มีอยู่

1. ไปที่ **แผนการทดสอบ \> แผนการทดสอบ** และเลือกแผน **แผนการทดสอบ RSAT**
2. เลือก **เพิ่มรายการที่มีอยู่**
3. ในหน้า **เพิ่มกรณีการทดสอบไปยังชุด** ให้เลือก **รันการสอบถาม**
4. เลือกกรณีการทดสอบใหม่ที่สร้างขึ้นสำหรับ **นำผลิตภัณฑ์ออกใช้** แล้วจากนั้น เลือก **เพิ่มกรณีการทดสอบ** ที่มุมล่างด้านล่างขวาของหน้า (ปุ่มนี้จะไม่แสดงในภาพประกอบต่อไปนี้)

    ![หน้าเพิ่มกรณีการทดสอบไปยังชุด](./media/setup_rsa_tool_100.png)

    ตอนนี้ชุดการทดสอบมีกรณีการทดสอบสองกรณี

    ![กรณีการทดสอบสองกรณีในชุดการทดสอบ](./media/setup_rsa_tool_101.png)

#### <a name="load-test-cases-into-rsat"></a>โหลดกรณีการทดสอบลงใน RSAT

1. เปิด RSAT แล้วเลือก **โหลด**
2. มีการโหลดกรณีการทดสอบ และคุณได้รับคำเตือนที่ระบุว่า "การดำเนินการนี้จะเขียนทับไฟล์ข้อมูลการทดสอบของ Excel การเปลี่ยนแปลงเฉพาะที่จะสูญหายไป คุณต้องการดำเนินการต่อหรือไม่?" เลือก **ใช่** เพื่อปรับปรุงไฟล์พารามิเตอร์ Excel ในระบบเฉพาะที่ แต่ไม่ใช่ไฟล์พารามิเตอร์ excel ที่ถูกอัปโหลดไปยัง Azure DevOps

    ![ข้อความเตือน](./media/setup_rsa_tool_102.png)

    มีการโหลดกรณีการทดสอบทั้งสองกรณี พร้อมกับไฟล์พารามิเตอร์ Excel สำหรับกรณีการทดสอบแรก เนื่องจากคุณเลือก **อัปโหลด** ในการรันครั้งล่าสุด ไฟล์พารามิเตอร์จะถูกดึงมาจาก Azure DevOps

    ![โหลดกรณีการทดสอบแล้ว](./media/setup_rsa_tool_103.png)

3. เลือกเฉพาะกรณีการทดสอบที่สอง แล้วจากนั้น เลือก **สร้าง \> สร้างการดำเนินการทดสอบและไฟล์พารามิเตอร์**

#### <a name="edit-the-excel-parameter-file"></a>แก้ไขไฟล์พารามิเตอร์ของ Excel

1. เลือกเฉพาะกรณีการทดสอบที่สอง แล้วจากนั้น เลือก **แก้ไข** เพื่อเปิดไฟล์พารามิเตอร์ Excel ที่สอดคล้องกัน
2. คัดลอกตัวแปรที่บันทึกไว้ **{{EcoResProductCreate\_รหัสประจำตัว\_ProductNumber\_คัดลอก}}** (ดูส่วน [ปรับเปลี่ยนการบันทึกงานที่มียู่เพื่อสร้างตัวแปรที่บันทึกไว้](#modify-an-existing-task-recording-to-create-a-saved-variable) ) จากกรณีการทดสอบแรก ลงในฟิลด์ทั้งหมดที่มีการใช้หมายเลขผลิตภัณฑ์ ในกรณีนี้ คุณคัดลอกตัวแปรไปยังฟิลด์ **หมายเลขผลิตภัณฑ์** และฟิลด์ **ตรวจสอบความถูกต้องของหมายเลขผลิตภัณฑ์** บนแผ่นงาน **EcoResProductListPage**

    ![ฟิลด์หมายเลขผลิตภัณฑ์และฟิลด์ตรวจสอบความถูกต้องของหมายเลขผลิตภัณฑ์](./media/setup_rsa_tool_104.png)

    > [!NOTE]
    > สามารถส่งผ่านตัวแปรระหว่างการทดสอบได้เฉพาะในระหว่างการรันการทดสอบเดียวกันเท่านั้น ชื่อของตัวแปรต้องตรงกัน

3. เลือก **บันทึก** และจากนั้น ปิดสมุดงาน Excel
4. เลือก **อัปโหลด** เพื่อบันทึกการเปลี่ยนแปลงที่คุณทำไว้กับไฟล์พารามิเตอร์ Excel

#### <a name="run-the-chained-test-cases"></a>รันกรณีทดสอบที่ถูกเชื่อมโยง

1. เลือกกรณีการทดสอบทั้งสองกรณี แล้วเลือก **รัน**
2. ตรวจสอบว่ากรณีการทดสอบทั้งสองกรณีผ่าน

    ![ฟิลด์ผลลัพธ์ที่ถูกตั้งค่าเป็น ผ่าน สำหรับกรณีการทดสอบทั้งสองกรณี](./media/setup_rsa_tool_105.png)
