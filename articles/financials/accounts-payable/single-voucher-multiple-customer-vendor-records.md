# <a name="single-voucher-with-multiple-customer-or-vendor-records"></a>ใบสำคัญเดียวที่มีเรกคอร์ดของลูกค้าหรือผู้จัดจำหน่ายหลายรายการ

[!include[banner](../includes/banner.md)]


หัวข้อนี้แสดงภาพรวมของสิ่งที่เกิดขึ้นเมื่อคุณลงรายการบัญชีใบสำคัญเดียวกับเรกคอร์ของลูกค้าหรือผู้จัดจำหน่ายหลายรายการ ฟังก์ชันนี้จะถูกยกเลิกในเวอร์ชันในอนาคตของ Microsoft Dynamics 365 for Finance and Operations ดังนั้น เราไม่แนะนำให้ใช้วิธีการลงรายการบัญชีนี้ เนื่องจากจะมีผลกระทบด้านการบัญชีต่อการประมวลผลการชำระเงิน 

ตัวอย่างทั่วไปที่มีการใช้ใบสำคัญหนึ่งใบสำหรับลูกค้าหรือผู้จัดจำหน่ายหลายรายการรวมถึงการโอนย้ายยอดดุลระหว่างลูกค้า และการรวมยอดดุลระหว่างลูกค้าและผู้จัดจำหน่ายในองค์กรเดียวกัน 

คุณสามารถป้อนใบสำคัญที่ประกอบด้วยลูกค้าหรือผู้จัดจำหน่ายอย่างน้อยหนึ่งรายได้โดยใช้วิธีต่อไปนี้:

-   การใช้สมุดรายวันที่มีการเลือกตัวเลือก **หมายเลขใบสำคัญหนึ่งหมายเลขเท่านั้น** ไว้เพื่อให้รวมทุกรายการที่เพิ่มลงในสมุดรายวันรวมอยู่ในใบสำคัญเดียวกัน
-   การใช้ใบสำคัญหลายรายการ ซึ่งไม่มีบัญชีแยกประเภทตรงข้าม กับลูกค้าหรือผู้จัดจำหน่ายอย่างน้อยหนึ่งราย
-   การป้อนใบสำคัญที่มีบัญชีและบัญชีตรงข้ามเป็น ผู้จัดจำหน่าย/ผู้จัดจำหน่าย ลูกค้า/ลูกค้า ผู้จัดจำหน่าย/ลูกค้า หรือลูกค้า/ผู้จัดจำหน่าย

หัวข้อนี้แสดงวิธีการประมวลผลการชำระเงินเมื่อมีการลงรายการบัญชีใบสำคัญหนึ่งใบกับเรกคอร์ดของลูกค้าหรือผู้จัดจำหน่ายหลายรายการ นอกจากนี้ หัวข้อนี้ยังแสดงวิธีการแก้ไขปัญหาเพื่อช่วยให้คุณเข้าใจวิธีการหลีกเลี่ยงการใช้ใบสำคัญหนึ่งใบกับลูกค้าหรือผู้จัดจำหน่ายหลายรายการ โดยเฉพาะอย่างยิ่ง มีตัวอย่างที่แสดงสถานการณ์การชำระเงินทั่วไปสองรายการที่ได้รับผลกระทบจากการใช้ใบสำคัญหนึ่งใบกับลูกค้าหรือผู้จัดจำหน่ายหลายรายการ:

-   การบัญชีส่วนลดเงินสด
-   การบัญชีการประเมินค่าใหม่

## <a name="how-does-settlement-impact-single-voucher-usage"></a>การชำระเงินมีผลกระทบต่อการใช้ใบสำคัญเดียวอย่างไร
เมื่อมีการลงรายการบัญชีใบสำคัญที่ประกอบด้วยเรกคอร์ดลูกค้าหรือผู้จัดจำหน่ายหลายรายการ ใบสำคัญการบัญชีเดียวจะถูกสร้างขึ้นโดยประกอบด้วยยอดดุลบัญชีลูกหนี้หรือบัญชีเจ้าหนี้หลายรายการ ในระหว่างกระบวนการชำระเงิน รายการการบัญชีเดิมจะถูกใช้ในการสร้างรายการการบัญชีสำหรับส่วนลดเงินสด กำไรและขาดทุนที่ยังไม่รับรู้ กำไรและขาดทุนที่รับรู้ และการปลดปล่อยบัญชีสรุปของเอกสารต้นฉบับ ตัวอย่างเช่น ถ้าส่วนลดเงินสดถูกใช้เมื่อชำระเงินให้กับผู้จัดจำหน่ายสำหรับใบแจ้งหนี้ การบัญชีส่วนลดเงินสดจะต้องลงรายการบัญชีในบัญชีแยกประเภทของบัญชีเจ้าหนี้จากใบแจ้งหนี้ต้นฉบับ ถ้าใบแจ้งหนี้เดิมถูกลงรายการบัญชีในใบสำคัญที่ประกอบด้วยเรกคอร์ดของผู้จัดจำหน่ายหลายรายการ ระบบจะสรุปข้อมูลการบัญชีเดิม ในกรณีนี้ เนื่องจากไม่มีวิธีการเข้าถึงรายการบัญชีโดยละเอียดสำหรับแต่ละธุรกรรมของผู้จัดจำหน่ายในใบสำคัญเดียว จึงไม่สามารถกำหนดได้ว่าผู้ใช้ตั้งใจจะพิจารณาส่วนลดเงินสดอย่างไร

## <a name="one-voucher-for-balance-transfers-and-netting-scenarios"></a>ใบสำคัญหนึ่งใบสำหรับการโอนย้ายยอดดุลและการรวมสถานการณ์จำลอง
สถานการณ์จำลองที่ใช้ทั่วไปสองรายการที่ใช้ใบสำคัญหนึ่งใบที่มีลูกค้าหรือผู้จัดจำหน่ายหลายรายการที่มีการโอนย้ายยอดดุลจากลูกค้า/ผู้จัดจำหน่ายหนึ่งรายการไปยังลูกค้า/ผู้จัดจำหน่ายอีกหนึ่งรายการ และการรวมลูกค้าและผู้จัดจำหน่ายที่มีองค์กรเดียวกัน ตัวอย่างสองรายการต่อไปนี้แสดงวิธีการที่ต้องการสำหรับการป้อนสถานการณ์จำลองเหล่านี้ใน Finance and Operations โดยเป็นทางเลือกในการป้อนในใบสำคัญหนึ่งใบ 

*การโอนย้ายยอดดุล* คือใบสำคัญหนึ่งใบที่มีลูกค้าหลายรายการ ที่ถูกป้อนเพื่อวัตถุประสงค์ในการโอนย้ายยอดดุลจากลูกค้ารายหนึ่งไปยังลูกค้ารายอื่น (เช่นเดียวกันสำหรับผู้จัดจำหน่าย) สถานการณ์จำลองนี้อาจเกิดขึ้นเมื่อความรับผิดชอบสำหรับการชำระใบแจ้งหนี้เปลี่ยนเป็นฝ่ายอื่น เช่น บริษัทรองที่เปลี่ยนความรับผิดชอบให้กับบริษัทแม่
