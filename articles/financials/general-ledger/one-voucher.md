# <a name="one-voucher"></a>ใบสำคัญหนึ่งใบ

[!include [banner](../includes/banner.md)]


## <a name="what-is-one-voucher"></a>"ใบสำคัญหนึ่งใบ" คืออะไร?

ฟังก์ชันที่มีอยู่สำหรับสมุดรายวันทางการเงิน (สมุดรายวันทั่วไป สมุดรายวันสินทรัพย์ถาวร สมุดรายวันการชำระเงินของผู้จัดจำหน่าย และอื่นๆ) ช่วยให้คุณสามารถป้อนธุรกรรมบัญชีแยกประเภทย่อยต่างๆ ได้ (ลูกค้า ผู้จำหน่าย สินทรัพย์ถาวร โครงการ และธนาคาร) ในบริบทของใบสำคัญใบเดียว Microsoft แสดงถึงฟังก์ชันนี้เป็น *ใบสำคัญหนึ่งใบ* คุณสามารถสร้างใบสำคัญใบเดียวได้โดยใช้หนึ่งในวิธีต่อไปนี้:

- ตั้งค่าชื่อสมุดรายวัน (**บัญชีแยกประเภททั่วไป** \> **การตั้งค่าสมุดรายวัน** \> **ชื่อสมุดรายวัน**) เพื่อให้ฟิลด์ **ใบสำคัญใหม่** ถูกกำหนดเป็น **หมายเลขใบสำคัญหนึ่งใบเท่านั้น** ทุกรายการที่คุณเพิ่มลงในสมุดรายวันจะถูกรวมอยู่ในใบสำคัญเดียวกัน ณ ขณะนี้ ดังนั้น จะมีการป้อนใบสำคัญเป็นใบสำคัญหลายรายการ เป็นบัญชี/บัญชีตรงข้ามในรายการเดียวกัน หรือการรวม

    [![รายการเดียว](./media/same-line.png)](./media/same-line.png)

    > [!IMPORTANT]
    > คำนิยามของ ใบสำคัญหนึ่งใบ **ไม่** ครอบคลุมกรณีที่ชื่อสมุดรายวันถูกตั้งค่าเป็น **หมายเลขใบสำคัญหนึ่งเท่านั้น** แต่ผู้ใช้จะป้อนใบสำคัญซึ่งรวมถึงชนิดบัญชีแยกประเภทเท่านั้น ในหัวข้อนี้ ใบสำคัญหนึ่งใบ หมายความว่ามีใบสำคัญเดียวที่ประกอบด้วยผู้จัดจำหน่าย ลูกค้า ธนาคาร สินทรัพย์ถาวร หรือโครงการ มากกว่าหนึ่งรายการ

- ป้อนใบสำคัญหลายรายการที่ซึ่งไม่มีบัญชีตรงข้าม

    [![ใบสำคัญหลายรายการ](./media/Multi-line.png)](./media/Multi-line.png)

- ป้อนใบสำคัญที่ทั้งบัญชีและบัญชีตรงข้ามประกอบด้วยชนิดบัญชีแยกประเภทย่อย เช่น **ผู้จัดจำหน่าย**/**ผู้จัดจำหน่าย** **ลูกค้า**/**ลูกค้า** **ผู้จัดจำหน่าย**/**ลูกค้า** หรือ **ธนาคาร**/**ธนาคาร**

    [![ใบสำคัญในบัญชีแยกประเภทย่อย](./media/subledger.png)](./media/subledger.png)

## <a name="issues-with-one-voucher"></a>ปัญหาเกี่ยวกับใบสำคัญหนึ่งใบ

ฟังก์ชันใบสำคัญหนึ่งใบทำให้เกิดปัญหาในระหว่างการชำระเงิน การคำนวณภาษี การกลับรายการธุรกรรม การกระทบยอดของบัญชีแยกประเภทย่อยไปยังบัญชีแยกประเภททั่วไป การรายงานทางการเงิน และอื่นๆ (สำหรับข้อมูลเพิ่มเติมเกี่ยวกับปัญหาที่อาจเกิดขึ้นระหว่างการชำระเงิน โปรดดู ตัวอย่างเช่น [ใบสำคัญเดียวที่มีเรกคอร์ดลูกค้าหรือผู้จัดจำหน่ายหลายราย](https://docs.microsoft.com/en-us/dynamics365/unified-operations/financials/accounts-payable/single-voucher-multiple-customer-vendor-records)) ในการทำงานและการรายงานอย่างถูกต้อง กระบวนการและรายงานเหล่านี้จำเป็นต้องมีรายละเอียดของธุรกรรม ถึงแม้ว่าบางสถานการณ์อาจยังคงทำงานได้อย่างถูกต้อง โดยขึ้นอยู่กับการตั้งค่าองค์กรของคุณ แต่มักจะมีปัญหาเมื่อมีการป้อนธุรกรรมต่างๆ ในใบสำคัญหนึ่งใบ

ตัวอย่างเช่น คุณลงรายการบัญชีใบสำคัญหลายรายการ

[![ตัวอย่าง](./media/example.png)](./media/example.png)

จากนั้น คุณสร้างรายงาน **ค่าใช้จ่ายโดยเรียงตามผู้จัดจำหน่าย** ในพื้นที่ทำงาน **ข้อมูลเชิงลึกทางการเงิน** ในรายงานนี้ มีการจัดกลุ่มยอดดุลบัญชีค่าใช้จ่ายโดยกลุ่มผู้จัดจำหน่าย และจากนั้น ผู้จัดจำหน่าย เมื่อมีการสร้างรายงาน ระบบจะไม่สามารถกำหนดได้ว่ากลุ่มของผู้จัดจำหน่าย/ผู้จัดจำหน่ายรายใดที่ทำให้เกิดค่าใช้จ่าย 250.00 เนื่องจากรายละเอียดธุรกรรมขาดหายไป ระบบจะถือว่าค่าใช้จ่ายทั้ง 250.00 ถูกทำให้เกิดโดยผู้จัดจำหน่ายแรกที่พบในใบสำคัญ ดังนั้นค่าใช้จ่าย 250.00 ซึ่งจะถูกรวมอยู่ในยอดดุลสำหรับบัญชีหลัก 600120 จะถูกแสดงอยู่ภายใต้กลุ่มผู้จัดจำหน่าย/ผู้จัดจำหน่ายนั้น อย่างไรก็ตาม เป็นไปได้อย่างมากที่ผู้จัดจำหน่ายแรกในใบสำคัญจะไม่ใช่ผู้จัดจำหน่ายที่ถูกต้อง ดังนั้น รายงานอาจไม่ถูกต้อง

## <a name="the-future-of-one-voucher"></a>อนาคตของใบสำคัญหนึ่งใบ

เนื่องจากปัญหาที่ระบุไว้ก่อนหน้า ฟังก์ชันใบสำคัญหนึ่งใบจะถูกทำให้ล้าสมัย อย่างไรก็ตาม เนื่องจากมีช่องว่างทางฟังก์ชันที่ขึ้นอยู่กับฟังก์ชันนี้ ฟังก์ชันจะไม่กลายเป็นล้าสมัยทั้งหมดพร้อมกัน จะมีการใช้กำหนดการต่อไปนี้แทน:

- **การนำออกใช้ในฤดูใบไม้ผลิ 2018** – ตามค่าเริ่มต้น ฟังก์ชันจะถูกปิดโดยค่าเริ่มต้นโดยใช้พารามิเตอร์ **อนุญาตธุรกรรมหลายรายการภายในใบสำคัญหนึ่งใบ** บนแท็บ **ทั่วไป** ของหน้า **พารามิเตอร์บัญชีแยกประเภททั่วไป** อย่างไรก็ตาม คุณสามารถเปิดฟังก์ชันได้ ถ้าองค์กรของคุณมีสถานการณ์สมมติที่อยู่ในหนึ่งในช่องว่างเชิงฟังก์ชันที่แสดงรายการในหัวข้อนี้ในภายหลัง

    - ถ้าลูกค้ามีสถานการณ์ทางธุรกิจที่ไม่ต้องการใบสำคัญหนึ่งใบ พวกเขาไม่ควรเปิดฟังก์ชัน Microsoft จะไม่แก้ไข "บัก" ในพื้นที่ที่ถูกระบุในหัวข้อนี้ในภายหลัง ถ้าการมีใช้ฟังก์ชันนี้ ถึงแม้ว่าจะมีโซลูชันอื่นอยู่
    - หยุดใช้ใบสำคัญหนึ่งสำหรับการรวมลงใน Microsoft Dynamics 365 for Finance and Operations ถ้าไม่ต้องการฟังก์ชันสำหรับหนึ่งในช่องว่างทางฟังก์ชัน

- **การนำออกใช้ในภายหลัง** – จะมีการเติมช่องว่างเชิงฟังก์ชันทั้งหมด **หลังจากที่เติมช่องว่างเชิงฟังก์ชัน และมีการจัดส่งคุณลักษณะใหม่ จะใช้เวลาอย่างน้อยหนึ่งปีก่อนที่จะการปิดมีฟังก์ชันใบสำคัญหนึ่งใบอย่างถาวร** เนื่องจากลูกค้าและผู้ขายซอฟต์แวร์อิสระ (ISVs) ต้องมีเวลาเพียงพอที่จะรับมือกับฟังก์ชันการทำงานใหม่ ตัวอย่างเช่น พวกเขาอาจต้องปรับปรุงกระบวนการทางธุรกิจ เอนทิตี และการรวมของพวกเขา

> [!IMPORTANT]
> ตัวเลือก **หมายเลขใบสำคัญหนึ่งใบเท่านั้น** **ไม่** ได้ถูกลบออกจากการตั้งค่าชื่อสมุดรายวัน ตัวเลือกนี้จะยังคงได้รับการสนับสนุน เมื่อใบสำคัญประกอบด้วยชนิดบัญชีแยกประเภทเท่านั้น ลูกค้าต้องใช้ความระมัดระวัง เมื่อพวกเขาใช้การตั้งค่านี้ เนื่องจากใบสำคัญจะไม่ถูกลงรายการบัญชี ถ้าพวกเขาใช้ตัวเลือก **หมายเลขใบสำคัญหนึ่งใบเท่านั้น** แต่จากนั้น ป้อนลูกค้า ผู้ขาย ธนาคาร สินทรัพย์ถาวร หรือโครงการ มากกว่าหนึ่งรายการ นอกจากนี้ ลูกค้ายังคงสามารถป้อนการผสมกันของชนิดบัญชีแยกประเภทย่อยได้ เช่น การชำระเงินในใบสำคัญเดียวที่ประกอบด้วยชนิดบัญชีของ **ผู้จัดจำหน่าย**/**ธนาคาร**

## <a name="why-use-one-voucher"></a>ทำไมจึงใช้ใบสำคัญหนึ่งใบ?

โดยขึ้นอยู่กับการสนทนากับลูกค้า Microsoft ได้คอมไพล์รายการสถานการณ์จำลองต่อไปนี้ที่ซึ่งลูกค้าใช้ฟังก์ชันใบสำคัญหนึ่งใบ หรือเหตุผลว่าทำไมพวกเขาจึงใช้ ความต้องการทางธุรกิจเหล่านี้บางอย่างสามารถตอบสนองได้โดยใช้ใบสำคัญหนึ่งใบเท่านั้น อย่างไรก็ตาม สำหรับหลายๆ สถานการณ์จำลอง ทางเลือกตอบสนองความต้องการทางธุรกิจเดียวกันได้

### <a name="scenarios-that-require-one-voucher"></a>สถานการณ์จำลองที่ต้องใช้ใบสำคัญหนึ่งใบ

สถานการณ์จำลองต่อไปนี้สามารถดำเนินการได้โดยใช้ฟังก์ชันใบสำคัญหนึ่งใบเท่านั้น ถ้าองค์กรของคุณมีสถานการณ์จำลองเหล่านี้ใดๆ คุณต้องเปิดใช้งานธุรกรรมต่างๆ ที่จะป้อนในใบสำคัญ โดยการเปลี่ยนการตั้งค่าของพารามิเตอร์ **อนุญาตธุรกรรมหลายรายการภายในใบสำคัญหนึ่งใบ** บนหน้า **พารามิเตอร์บัญชีแยกประเภททั่วไป** ช่องว่างเชิงฟังก์ชันเหล่านี้จะถูกเติมข้อมูลโดยใช้ลักษณะการทำงานอื่นๆ ในการนำออกใช้ในภายหลัง

### <a name="post-vendor-or-customer-payments-in-summary-form-to-a-bank-account"></a>ลงรายการบัญชีการชำระเงินของผู้จัดจำหน่ายหรือลูกค้าในแบบฟอร์มสรุปไปยังบัญชีธนาคาร

**สถานการณ์จำลอง** องค์กรสื่อสารรายการของผู้จัดจำหน่ายและยอดเงินไปยังธนาคาร และธนาคารใช้รายการนี้เพื่อชำระแก่ผู้จัดจำหน่ายในนามขององค์กร ธนาคารลงรายการบัญชีผลรวมของการชำระเงินเป็นการเบิกถอนเดียวในบัญชีธนาคาร

แบบการสรุปของการชำระเงินให้แก่ผู้จัดจำหน่ายได้รับการสนับสนุนผ่านทางใบสำคัญหนึ่งใบเท่านั้น ผู้จัดจำหน่ายแต่ละรายจะถูกป้อนเป็นรายการแยกต่างหากเพื่อรักษารายละเอียดในการบัญชีเจ้าหนี้บัญชีแยกประเภทย่อย อย่างไรก็ตาม ยอดเงินสรุปสำหรับการชำระเงินทั้งหมดถูกออฟเซ็ตไปยังรายการเดียวสำหรับบัญชีธนาคาร ดังนั้น การถอนจะถูกแสดงขึ้นเป็นยอดเงินสรุปเดียวในบัญชีแยกประเภทย่อยของธนาคาร

**สถานการณ์จำลอง** องค์กรฝากการชำระเงินของลูกค้า หรือธนาคารฝากการชำระเงินของลูกค้าในนามขององค์กร และการฝากเงินถูกแสดงเป็นจำนวนเงินรวมทั้งหมดในบัญชีธนาคาร

แบบการสรุปของการชำระเงินของลูกค้าโดยทั่วไปจะได้รับการสนับสนุนผ่านทางฟังก์ชันใบนำฝาก อย่างไรก็ตาม ถ้าคุณกำลังใช้ "ระหว่างกาล" ในวิธีของการชำระเงิน สถานการณ์นี้ได้รับการสนับสนุนผ่านทางใบสำคัญหนึ่งใบเท่านั้น มีการป้อนการชำระเงินของลูกค้าในลักษณะเดียวกับที่ระบุไว้สำหรับแบบการสรุปการชำระเงินของผู้จัดจำหน่าย

### <a name="mechanism-to-group-transactions-from-a-business-event"></a>กลไกในการจัดกลุ่มธุรกรรมจากเหตุการณ์ทางธุรกิจ

**สถานการณ์จำลอง** องค์กรมีเหตุการณ์ทางธุรกิจเดียวที่ก่อให้เกิดธุรกรรมที่หลากหลาย อย่างไรก็ตาม แผนกการบัญชีต้องการดูรายการการบัญชีพร้อมๆ กันเพื่อความสามารถในการตรวจสอบที่ดีขึ้น

หากองค์กรต้องดูรายการการบัญชีจากเหตุการณ์ทางธุรกิจพร้อมๆ กัน จะต้องใช้ใบสำคัญหนึ่งใบ 

### <a name="country-specific-features"></a>ลักษณะการทำงานเฉพาะประเทศ

**สถานการณ์จำลอง** ในขณะนี้ คุณลักษณะ Single Administrative Document (SAD) สำหรับโปแลนด์ ต้องให้มีการใช้ใบสำคัญเดียว จนกว่าตัวเลือกการจัดกลุ่มจะพร้อมใช้งานสำหรับคุณลักษณะนี้ คุณต้องใช้ฟังก์ชันใบสำคัญหนึ่งใบต่อ อาจมีคุณลักษณะเฉพาะประเทศเพิ่มเติมที่ต้องใช้ฟังก์ชันใบสำคัญหนึ่งใบ

### <a name="customer-prepayment-payment-journal-that-has-taxes-on-multiple-lines"></a>สมุดรายวันการชำระเงินของการชำระเงินล่วงหน้าของลูกค้าที่มีภาษีใน "รายการ" ที่หลากหลาย

ในสถานการณ์จำลองนี้ ลูกค้าในใบสำคัญเดียวเป็นลูกค้าเดียวกัน เนื่องจากธุรกรรมจำลองรายการของใบสั่งของลูกค้า ต้องป้อนการชำระเงินล่วงหน้าในใบสำคัญเดียว เนื่องจากต้องทำการคำนวณภาษีใน "รายการ" ในการชำระเงินเดียวที่ลูกค้าจ่ายไว้

### <a name="customer-reimbursement"></a>การชำระเงินคืนของลูกค้า

**สถานการณ์จำลอง** ลูกค้าทำการชำระเงินล่วงหน้าสำหรับใบสั่ง และรายการของใบสั่งมีภาษีอื่นที่ต้องถูกบันทึกสำหรับการชำระเงินล่วงหน้า การชำระเงินของลูกค้าของการชำระเงินล่วงหน้าเป็นธุรกรรมหนึ่งที่จำลองรายการของใบสั่ง เพื่อให้สามารถบันทึกภาษีที่เหมาะสมสำหรับยอดเงินในแต่ละรายการได้

ถ้ามีการรันงานประจำงวดการชำระคืนเงินจากโมดูลบัญชีลูกหนี้ จะสร้างธุรกรรมเพื่อย้ายยอดดุลจากลูกค้าไปยังผู้จัดจำหน่าย สำหรับสถานการณ์จำลองนี้ ใบสำคัญหนึ่งต้องใช้ในการชำระเงินคืนลูกค้า

### <a name="fixed-asset-maintenance-catch-up-depreciation-split-asset-calculate-depreciation-on-disposal"></a>การบำรุงรักษาสินทรัพย์ถาวร: ทำให้ค่าเสื่อมราคาตรงกับปัจจุบัน แบ่งสินทรัพย์ คำนวณค่าเสื่อมราคาในการขายทิ้ง
ธุรกรรมสินทรัพย์ถาวรต่อไปนี้ยังจะสร้างธุรกรรมแบบหลายรายการในใบสำคัญเดียวอีกด้วย:

- มีการดำเนินการซื้อเพิ่มเติมในสินทรัพย์ และมีคำนวณค่าเสื่อมราคาที่ 'ตรงกับปัจจุบัน'
- มีการแบ่งสินทรัพย์
- มีการเปิดพารามิเตอร์เพื่อคำนวณค่าเสื่อมราคาในการขายทิ้ง และจากนั้น จะมีการตัดจำหน่ายสินทรัพย์
- วันที่เริ่มใช้งานของสินทรัพย์เป็นวันก่อนหน้าวันที่ซื้อสินทรัพย์ ดังนั้น การปรับปรุงค่าเสื่อมราคาจะถูกลงรายการบัญชี

### <a name="bills-of-exchange-and-promissory-notes"></a> ตั๋วแลกเงินและตั๋วสัญญาใช้เงิน
ตั๋วแลกเงินและตั๋วสัญญาใช้เงินต้องมีใบสำคัญหนึ่งที่จะถูกใช้ เนื่องจากธุรกรรมย้ายลูกค้าหรือยอดดุลผู้จัดจำหน่ายจากบัญชีแยกประเภทของบัญชีลูกหนี้/บัญชีบัญชีเจ้าหนี้หนึ่ง ไปยังอีกแห่งหนึ่ง ตามสถานะของการชำระเงิน

## <a name="scenarios-that-dont-require-one-voucher"></a>สถานการณ์จำลองที่ไม่ต้องใช้ใบสำคัญหนึ่งใบ

สถานการณ์จำลองต่อไปนี้สามารถทำให้สำเร็จได้โดยวิธีอื่นๆ และไม่ควรใช้ฟังก์ชันใบสำคัญหนึ่งใบ

### <a name="post-customer-payments-in-summary-form-to-the-bank-account"></a>ลงรายการบัญชีการชำระเงินของลูกค้าในแบบฟอร์มสรุปไปยังบัญชีธนาคาร

องค์กรฝากการชำระเงินของลูกค้า หรือธนาคารฝากการชำระเงินของลูกค้าในนามขององค์กร และการฝากเงินจะแสดงเป็นจำนวนเงินรวมทั้งหมดในบัญชีธนาคาร

การสรุปของการชำระเงินของลูกค้าได้รับการสนับสนุนผ่านทางฟังก์ชันการฝาก เมื่อ "การเชื่อมโยง" ไม่ได้ถูกใช้ในวิธีการชำระเงิน

### <a name="netting"></a>การหักกลบลบหนี้

ในการหักกลบลบหนี้ ยอดดุลสำหรับผู้จัดจำหน่ายและลูกค้าจะถูกรวมระหว่างกัน เนื่องจากผู้จัดจำหน่ายและลูกค้าเป็นฝ่ายเดียวกัน วิธีการนี้ลดขนาดการแลกเปลี่ยนของเงินระหว่างองค์กรและฝ่ายลูกค้า/ผู้จัดจำหน่าย

สามารถทำการหักกลบลบหนี้ให้สำเร็จได้โดยการป้อนการเพิ่มขึ้นและการลดลงในใบสำคัญที่แยกต่างหาก และจากนั้นลงรายการบัญชีออฟเซ็ตไปยังบัญชีแยกประเภทการหักบัญชี

### <a name="post-in-summary-to-the-general-ledger"></a>ลงรายการบัญชีในสรุปไปยังบัญชีแยกประเภททั่วไป

องค์กรมักจะต้องการลงรายการบัญชีไปยังบัญชีแยกประเภททั่วไปในฟอร์มสรุปเพื่อลดจำนวนของข้อมูล อย่างไรก็ตาม โดยปกติ องค์กรเหล่านั้นยังคงจำเป็นต้องรักษารายละเอียดของธุรกรรม เมื่อทำการลงรายการบัญชีในฟอร์มสรุปโดยใช้ใบสำคัญเดียวแล้ว จะไม่รู้จักและไม่สามารถรักษารายละเอียดของธุรกรรมได้

- เนื่องจากไม่สามารถรักษารายละเอียดของธุรกรรมได้ในขณะนี้ เราขอแนะนำให้องค์กร **ไม่** ต้องใช้ใบสำคัญหนึ่งใบเพื่อลงรายการบัญชีในฟอร์มสรุป
- หลังจากการสนับสนุนสำหรับใบสำคัญหนึ่งใบถูกลบออก สามารถใช้เอกสารต้นทางและกรอบงานการบัญชีในสมุดรายวันได้ จากนั้น กรอบงานเหล่านี้จะรักษารายละเอียดธุรกรรมและสนับสนุนการสรุปลงในบัญชีแยกประเภททั่วไป


### <a name="settle-multiple-unposted-payments-to-the-same-invoice"></a>จับคู่การชำระเงินที่ไม่ได้ลงรายการบัญชีแบบหลายรายการกับใบแจ้งหนี้เดียวกัน

สถานการณ์นี้มักจะพบในองค์กรขายปลีกที่ลูกค้าสามารถใช้วิธีการชำระเงินที่หลากหลายเพื่อชำระเงินสำหรับการซื้อ ในสถานการณ์จำลองนี้ องค์กรต้องสามารถบันทึกการชำระเงินที่ไม่ได้ลงรายการบัญชีแบบหลายรายการ และจับคู่รายการเหล่านั้นกับใบแจ้งหนี้ของลูกค้าได้

คุณลักษณะใหม่ที่ถูกเพิ่มใน Microsoft Dynamics 365 for Operations รุ่น 1611 (พฤศจิกายน 2016) เปิดใช้งานการชำระเงินที่ไม่ได้ลงรายการบัญชีที่หลากหลายให้ถูกชำระเงินเทียบกับใบแจ้งหนี้เดียว การชำระเงินของลูกค้าที่หลากหลายไม่ต้องถูกป้อนในใบสำคัญเดียวอีกต่อไป

### <a name="import-bank-statement-transactions"></a>นำเข้าธุรกรรมใบแจ้งยอดจากธนาคาร

ธนาคารมักจะชำระค่าจ้างและได้รับการชำระเงินในนามขององค์กร และธุรกรรมเหล่านั้นจะถูกบันทึกไว้ใน Finance and Operations โดยใช้ไฟล์ที่ได้รับจากธนาคาร องค์กรมักจะต้องการจัดกลุ่มธุรกรรมดังกล่าวเข้าด้วยกัน โดยใช้หมายเลขใบแจ้งยอดจากธนาคารในไฟล์ เนื่องจากแต่ละธุรกรรมจะปรากฏในรายละเอียดในใบแจ้งยอดจากธนาคาร ไม่จำเป็นต้องมีแบบการสรุปในบัญชีแยกประเภทย่อยจากธนาคาร

สามารถจัดกลุ่มธุรกรรมได้โดยใช้ฟิลด์อื่นๆ บนสมุดรายวัน เช่น หมายเลขชุดงานสมุดรายวันเอง หรือหมายเลขเอกสาร


### <a name="transfer-balances"></a>โอนย้ายยอดดุล

องค์กรอาจจำเป็นต้องโอนย้ายยอดดุลจากผู้จัดจำหน่ายหนึ่งไปยังอีกผู้จัดจำหน่ายหนึ่ง เพราะข้อผิดพลาดหรือเนื่องจากอีกผู้จัดจำหน่ายหนึ่งได้ควบคุมหนี้สินแล้ว การโอนย้ายของชนิดนี้ยังเกิดขึ้นสำหรับชนิดบัญชีอีกต่างๆ อีกด้วย เช่น **ลูกค้า** และ **ธนาคาร**

การโอนย้ายยอดดุลจากบัญชีหนึ่ง (ผู้จัดจำหน่าย ลูกค้า ธนาคาร และอื่นๆ) ไปยังบัญชีอื่น สามารถทำได้โดยใช้ใบสำคัญแยกต่างหาก และสามารถลงรายการบัญชีออฟเซ็ตไปยังบัญชีแยกประเภทการหักบัญชีได้อีกด้วย

### <a name="enter-beginning-balances"></a>ป้อนยอดดุลต้นงวด

องค์กรมักจะป้อนยอดดุลต้นงวดสำหรับบัญชีแยกประเภทย่อย (ผู้จัดจำหน่าย ลูกค้า สินทรัพย์ถาวร และอื่นๆ) เป็นธุรกรรมใบสำคัญหนึ่งใบ ยอดดุลต้นงวดสำหรับแต่ละบัญชีแยกประเภทย่อย สามารถถูกป้อนเป็นใบสำคัญแยกต่างหาก และสามารถลงรายการบัญชีออฟเซ็ตไปยังบัญชีแยกประเภทการหักบัญชีได้

### <a name="correct-the-accounting-entry-of-a-posted-customer-or-vendor-document"></a>แก้ไขรายการการบัญชีของเอกสารลูกค้าหรือผู้จัดจำหน่ายที่ลงรายการบัญชี

องค์กรอาจจำเป็นต้องแก้ไขบัญชีแยกประเภทของบัญชีลูกหนี้หรือบัญชีเจ้าหนี้บัญชี สำหรับรายการการบัญชีของใบแจ้งหนี้ที่ลงรายการบัญชี แต่ไม่สามารถกลับรายการหรือแก้ไขใบแจ้งหนี้โดยใช้กลไกอื่นๆ ได้

ถ้าต้องทำการแก้ไขไปยังบัญชีลูกหนี้ หรือแยกประเภทของบัญชีเจ้าหนี้บัญชี ต้องทำการปรับปรุงโดยตรงไปยังบัญชีแยกประเภท ไม่สามารถทำการปรับปรุงได้โดยการลงรายการบัญชีโดยใช้ผู้จัดจำหน่ายหรือลูกค้า วิธีการนี้จำเป็นต้องทำการปรับปรุงในระหว่าง "เวลาการหยุดทำงาน" เพื่อให้บัญชีแยกประเภทสามารถป้อนข้อมูลด้วยตนเองชั่วคราว

### <a name="the-system-allows-it"></a>"ระบบอนุญาต"
